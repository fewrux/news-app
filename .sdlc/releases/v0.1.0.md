## Rollback plan

If a regression surfaces post-merge:

1. **Revert the squash commit on `main`** via PR (`git revert <squash-sha>`
   on a `hotfix/revert-pr-7` branch → PR → squash-merge). Trunk-based
   discipline applies; no force-push to `main` per
   `.cursor/rules/branch-discipline.mdc`.
2. **No database to back out.** No schema migration, no data writes.
3. **No SLO to defend.** App surface unchanged; Vercel will serve the
   reverted SDLC files identically to end users.
4. **Re-open `HANDOFF-2026-05-28-sdlc-doctor`** if it was moved (it
   stays `open` after this release by design, so no action needed there).
5. **File `/incident`** documenting which sub-system (handoff queue,
   tracker adapter, doctor card, YAML fix) caused the rollback and what
   the next-session implementer needs to know.

Rollback risk for this release is low — there is no live failure mode
visible to end users; only agents/maintainer interacting with `.sdlc/`
and `.cursor/` surfaces can be affected.
