## Findings

### WARN — doctor-eval spurious exit-1 side effect

**Rule applied:** General software quality; does not break any spec gate.

`scripts/sdlc-doctor.mjs` calls `main()` at module scope (line 986) without
an entry-point guard. When `doctor-eval.mjs` imports `{ CHECK_IDS }` from
the doctor, the async `main()` launches as an unwaited side effect. After the
eval's own `main()` completes (all cases pass, "All N cases passed." printed),
the doctor's async `main()` resolves, writes JSON + summary to stdout/stderr,
and calls `exit(1)` (because the live tree has 4 fail findings). This makes
`npm run doctor:eval` exit 1 even when all assertions pass.

**Impact:** No CI workflow currently invokes `doctor-eval.mjs`, so no gate
breaks today. But it makes the eval misleading to run locally and would fail
any future CI step added for AC-3.

**Recommendation:** Guard the entry-point call:
```js
import { fileURLToPath } from "node:url";
if (process.argv[1] === fileURLToPath(import.meta.url)) {
  main();
}
```
**Severity:** `warn` (non-blocking; no AC-13 or CI gate failure)

### INFO — artifact.legacy-plane-issue broader than spec description

AC-11 specifies INT-0001 as the canonical example. The script correctly flags
INT-0001, INT-0002, INT-0003, and INT-0004 — all of which carry `plane_issue:`
fields. The doctor-2026-05-28.md report mentions only INT-0001 in its findings
table. Minor documentation gap; does not affect runtime behavior.

**Severity:** `info` (cosmetic)
